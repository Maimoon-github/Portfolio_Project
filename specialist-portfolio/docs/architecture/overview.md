## 1. Guiding Principles

This codebase is built on five non-negotiable rules derived from the bulletproof-react pattern:

| # | Principle | Rule |
|---|---|---|
| 1 | **Unidirectionality** | Code flows downward through defined layers. A layer may only import from layers below it — never sideways or upward. |
| 2 | **Single source of truth** | Design tokens live in `styles/variables.css`, shared interfaces live in `src/types/`, routes live in `src/config/routes.ts`. There is one definition of each concept. |
| 3 | **Explicit over implicit** | No magic strings. No hardcoded hex values in components. No inline route paths in `<Link>`. Everything is named, typed, and imported from a canonical location. |
| 4 | **Co-located concerns** | Each component owns its template (`.tsx`), its styles (`.module.css`), its prop types (`.types.ts`), and its tests (`.test.tsx`) in a single folder. |
| 5 | **Type safety at every boundary** | TypeScript strict mode is non-negotiable. Props, data schemas, API responses, hook return values, and route params are all typed. Runtime surprises are compile-time errors. |

## 2. Architectural Decisions

### 2.1 CSS Modules vs. CSS-in-JS

**Decision:** CSS Modules.

**Rationale:**

CSS-in-JS libraries (styled-components, Emotion) process styles at runtime. When a component renders, the library serialises JavaScript template literals into plain CSS strings, injects them into `<style>` tags in the document `<head>`, and tracks which rules are active. This mechanism introduces measurable costs:

- **Runtime CPU overhead.** Every render that involves a style recalculation consumes CPU cycles for serialisation. Benchmark data shows styled-components adding 20–35ms of extra blocking time per page in component-heavy views.
- **Bundle size inflation.** CSS-in-JS runtime libraries add 15–45KB of JavaScript that must be downloaded, parsed, and executed before the application becomes interactive — a direct hit to Time to Interactive (TTI).
- **React DevTools noise.** Wrapper components generated by styled-components pollute the component tree, making debugging significantly harder.
- **SSR complexity.** Server-side rendering with CSS-in-JS requires careful critical-CSS extraction setup to prevent a flash of unstyled content (FOUC). CSS Modules require no such configuration.

CSS Modules have **zero runtime overhead**. All class name scoping and composition happens at build time via Vite's CSS pipeline. The output is a static `.css` file the browser caches independently from the JavaScript bundle. Applications with 500+ components show 20–35% better Lighthouse performance scores using CSS Modules over CSS-in-JS.

This project's constraint (no CSS-in-JS, prefer vanilla CSS) also aligns with this decision. CSS custom properties — defined in `src/styles/variables.css` — handle all theming needs that would otherwise require dynamic style injection.

The only capability CSS Modules cannot match is runtime dynamic styling (generating entirely new CSS rules on the fly based on user input). This project has no such requirement. All "dynamic" styling is expressed as state-driven class name switching between pre-defined CSS Module classes.

### 2.2 Typed Data (.ts) vs. JSON

**Decision:** Data files use `.ts` with explicit `const` arrays typed against interfaces from `src/types/`.

**Rationale:**

JSON files are inert. TypeScript cannot validate their contents against an interface at compile time — it can only infer a generic shape on import. A typo in a JSON value, a missing required field, or an incorrectly typed property silently passes the compiler and fails at runtime when the component tries to render it.

TypeScript data files (`.ts`) solve this with zero additional tooling:

```ts
// src/data/projects.ts
import type { Project } from '@/types';

export const projects: Project[] = [
  {
    id: 'rag-pipeline-v2',
    title: 'RAG Pipeline v2',
    status: 'Active',           // ✅ Compiler enforces ProjectStatus enum
    category: 'AIEngineering',  // ✅ Compiler enforces ProjectCategory enum
    metrics: [
      { icon: 'cpu', value: 94.7, label: 'Retrieval Precision (%)' }
    ],
    // Omitting a required field here is a compile error, not a runtime crash
  },
];
```

Benefits over JSON:

| Benefit | JSON | `.ts` with type annotation |
|---|---|---|
| Compile-time field validation | ❌ | ✅ |
| IDE autocomplete for values | ❌ | ✅ |
| Rename-refactoring (e.g., renaming `status` field) | ❌ grep only | ✅ Language server updates all references |
| Enum enforcement | ❌ | ✅ |
| Computed/derived fields | ❌ | ✅ |
| Tree-shakeable (unused entries eliminated from bundle) | ❌ | ✅ |

The `src/data/schemas/` directory contains Zod validation schemas for the same types. These are used when data might arrive from an external source (a future CMS, API, or `.md` frontmatter) and needs runtime validation in addition to compile-time type checking.

### 2.3 Centralised Types Directory

**Decision:** All shared TypeScript interfaces and type aliases live in `src/types/`, not co-located inside component folders.

**Rationale:**

Consider a type like `Project`. It is referenced in:

- `src/data/projects.ts` — data definition
- `src/components/ui/ProjectCard/ProjectCard.types.ts` — prop definition
- `src/pages/Portfolio/Portfolio.tsx` — filtering and rendering
- `src/pages/Projects/Projects.tsx` — archive rendering
- `src/pages/ProjectDetail/ProjectDetail.tsx` — detail view
- `src/hooks/useFilteredProjects.ts` — filtering logic
- `src/services/analytics.service.ts` — event tracking

If `Project` were defined inside `ProjectCard.types.ts`, every other file above would need to import from a component — a violation of the unidirectional flow rule and a source of circular dependency errors.

A centralised `src/types/` directory is the single source of truth:

```
src/types/
├── project.types.ts    → Project, ProjectMetric, ProjectStatus, ProjectCategory
├── blog.types.ts       → BlogPost, BlogCategory
├── tool.types.ts       → Tool, ToolCategory
├── resume.types.ts     → ResumeEntry, Certification, Skill
├── navigation.types.ts → NavItem, NavDropdown
├── common.types.ts     → SEOMeta, FilterOption, WithChildren, Breakpoint
└── index.ts            → barrel re-export of all types
```

Component folders may have their own `ComponentName.types.ts` files, but these contain **only** component-specific prop interfaces that extend or compose the shared types. They do not define domain models.

**ESLint enforcement:** The `import/no-restricted-paths` rule prevents any file in `src/types/` from importing from `src/components/`, `src/pages/`, or `src/services/`. Types are a leaf node in the dependency graph.

### 2.4 Services Layer

**Decision:** All external integration logic (form submission, analytics, future API calls) lives in `src/services/`.

**Rationale:**

Without a services layer, integration logic ends up inline inside components:

```tsx
// ❌ Without services layer — untestable, tightly coupled
function Contact() {
  const handleSubmit = async (data) => {
    const res = await fetch('/api/contact', {
      method: 'POST',
      body: JSON.stringify(data),
    });
    if (!res.ok) throw new Error('Submission failed');
  };
}
```

This pattern has three problems. First, the component is hard to unit test — you must mock `fetch` globally or render the full component in a simulated browser environment just to test the form logic. Second, if the API endpoint changes, you grep for the URL string across component files. Third, the error handling, retry logic, and response parsing are duplicated wherever similar calls are made.

The services layer solves all three:

```ts
// src/services/contact.service.ts
export async function submitContactForm(data: ContactFormData): Promise<ContactResponse> {
  const res = await fetch(import.meta.env.VITE_CONTACT_ENDPOINT, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data),
  });
  if (!res.ok) throw new ContactSubmissionError(res.status);
  return res.json() as Promise<ContactResponse>;
}
```

The component becomes a thin orchestrator:

```tsx
// ✅ With services layer — testable, decoupled
function Contact() {
  const handleSubmit = async (data: ContactFormData) => {
    await submitContactForm(data); // one import, fully typed, independently testable
  };
}
```

Unit tests for `contact.service.ts` mock `fetch` in isolation — no component rendering required. Swapping the contact backend means editing one file in `src/services/`.

### 2.5 Custom Hooks Directory

**Decision:** All reusable stateful logic lives in `src/hooks/`.

**Rationale:**

Custom hooks extract the *behaviour* of a component from its *template*. They are the mechanism by which complex, shared logic is written once and used in many places without prop drilling or context over-engineering.

This project uses custom hooks for:

| Hook | Extracted behaviour | Used by |
|---|---|---|
| `useScrollPosition` | Passive scroll event listener with throttling | `Navbar`, back-to-top button |
| `useFilteredProjects` | Filter and sort logic for project arrays | `Portfolio`, `Projects` |
| `useReducedMotion` | `prefers-reduced-motion` media query listener | Any animated component |
| `useTheme` | Theme state, `localStorage` persistence, `data-theme` DOM attribute | `App`, `ThemeToggle` |

Without `src/hooks/`, this logic would either be duplicated across page components, lifted into a context that is overkill for its scope, or left inline inside a single component that cannot share it.

Hooks in `src/hooks/` follow two rules:
1. They must be genuinely reused by two or more consumers, or sufficiently complex to warrant extraction for testability.
2. They must not import from `src/components/` or `src/pages/`. Hooks are shared utilities — they sit below the component layer in the dependency graph.

## 3. Unidirectional Data Flow

Code in this project flows in one direction. A module may import from layers below it in this hierarchy, but **never from layers above it or sideways across the same layer**:

```
┌─────────────────────────────────────────────┐
│  src/types/                                 │  Layer 0 — Domain contracts
│  Pure interfaces and enums. No imports.    │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│  src/data/                                  │  Layer 1 — Static data
│  Typed arrays conforming to types/.         │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│  src/config/ · src/utils/ · src/hooks/     │  Layer 2 — Shared logic
│  Routes, constants, pure functions,        │
│  reusable stateful hooks.                  │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│  src/services/                              │  Layer 3 — External integrations
│  API calls, analytics, form handlers.      │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│  src/components/                            │  Layer 4 — UI building blocks
│  ui/ primitives and layout/ wrappers.      │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│  src/pages/                                 │  Layer 5 — Route views
│  Compose components. Call hooks and        │
│  services. Never imported by peers.        │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│  src/App.tsx                                │  Layer 6 — Application root
│  Routing, global providers, Layout.        │
└─────────────────────────────────────────────┘
```

**ESLint enforcement** via `import/no-restricted-paths`:
- `src/types/` cannot import from any other `src/` subdirectory.
- `src/components/ui/` cannot import from `src/pages/`.
- `src/data/` cannot import from `src/components/` or `src/pages/`.
- No cross-page imports (`pages/Portfolio` cannot import from `pages/Projects`).

## 4. Folder Structure Reference

```
src/
├── assets/            Static resources (fonts, images, icons)
├── components/
│   ├── ui/            Design system primitives — stateless, reusable
│   └── layout/        Structural shell components (Layout, PageWrapper, SectionContainer)
├── config/            Typed application configuration (routes, navigation, SEO)
├── data/              Typed static data arrays + Zod schemas
├── hooks/             Reusable custom React hooks
├── pages/             Route-level view components
├── services/          External integration functions (API, analytics)
├── styles/            Global CSS, design tokens, theme overrides
├── tests/             Vitest setup, mocks
├── types/             Shared TypeScript interfaces and enums
└── utils/             Pure utility functions
```

### `src/components/ui/`

Houses all design system primitives: `Button`, `Card`, `Badge`, `ProjectCard`, `ToolCard`, `DataTable`, `FormInput`, `FilterBar`, `HeroKPIStrip`, `Navigation`, `Footer`.

These components are **stateless by default**. They receive data via typed props and emit events via typed callbacks. They have no knowledge of routing, data fetching, or application state. They are the leaves of the component tree.

Each component in `ui/` is documented, tested, and independently usable — equivalent to a local component library. Adding a new primitive to the design system means adding a new folder here. Nothing else changes.

### `src/components/layout/`

Houses components that define the structural shell of the application: `Layout` (header + main + footer wrapper), `PageWrapper` (12-column grid, 1280px max-width, 24px gutters), `SectionContainer` (vertical rhythm, accessible landmark, anchor ID).

These components know about layout constraints but not about page-level content. They accept `children` and optional configuration props and are used by every page without modification.

Separating `ui/` from `layout/` prevents structural shell logic (grid, max-width, landmark roles) from leaking into design system primitives, and prevents design system primitives from assuming anything about their placement context.

### `src/pages/`

Houses route-level components — one folder per URL route. Pages are "smart": they select data from `src/data/`, call hooks from `src/hooks/`, compose components from `src/components/`, and call services from `src/services/`.

Pages are **never imported by other pages or components**. They are only referenced in `App.tsx` route definitions. This is the top of the component tree; nothing imports from here.

Complex pages (e.g., `Home/`) extract their sections into a co-located `sections/` subdirectory to keep the root page component under 150 lines.

## 5. Component Folder Convention

Every component lives in its own folder. The folder name is the component's public name in PascalCase. The folder contains exactly these files:

```
Button/
├── Button.tsx           — Component implementation
├── Button.module.css    — Scoped styles (CSS Modules)
├── Button.types.ts      — Prop interfaces and local type aliases
├── Button.test.tsx      — Unit tests (Vitest + React Testing Library)
└── index.ts             — Public API barrel export
```

**Why this structure:**

- **One folder = one component** makes the file system self-documenting. `ls src/components/ui/` tells you exactly what design system primitives exist.
- **Co-location** means styles, types, and tests travel with the component. Deleting a component means deleting one folder — no orphaned CSS files, no lingering test files in a parallel directory.
- **No barrel file at the `ui/` directory level** — each component's `index.ts` is the barrel. Consumers import from the component folder: `import { Button } from '@/components/ui/Button'`.

**File responsibilities:**

| File | Contains | Does NOT contain |
|---|---|---|
| `Button.tsx` | JSX template, internal state if needed, event handler wiring | Business logic, API calls, hardcoded style values |
| `Button.module.css` | All visual styles using CSS variables from `tokens.css` | Hardcoded colour hex values, `!important` overrides |
| `Button.types.ts` | `ButtonProps`, `ButtonVariant`, `ButtonSize` — interfaces for this component only | Domain model types (those belong in `src/types/`) |
| `Button.test.tsx` | Render tests, interaction tests, accessibility assertions | Integration tests that cross component boundaries |
| `index.ts` | `export { Button } from './Button'` and type re-exports | Any implementation logic |

## 6. Barrel Export Pattern

Every folder that exposes a public API contains an `index.ts` file that re-exports its contents.

**Without barrel exports:**

```ts
import { Button } from '@/components/ui/Button/Button';
import type { ButtonProps } from '@/components/ui/Button/Button.types';
import { ProjectCard } from '@/components/ui/ProjectCard/ProjectCard';
```

**With barrel exports:**

```ts
import { Button } from '@/components/ui/Button';
import type { ButtonProps } from '@/components/ui/Button';
import { ProjectCard } from '@/components/ui/ProjectCard';
```

Barrel exports decouple the consumer from the internal file structure of the module. If `Button.tsx` is split into `Button.tsx` and `ButtonIcon.tsx`, no consumer import path changes — only `index.ts` is updated.

**`src/types/index.ts`** is the most important barrel in the project. It re-exports every shared interface, enabling:

```ts
import type { Project, BlogPost, Tool, NavItem } from '@/types';
```

Instead of four separate imports from four different files.

**Caution:** A barrel file at the `src/components/ui/` directory level (one `index.ts` re-exporting all components) is intentionally avoided. Such a file causes Vite/Rollup to include all components in the initial chunk even when only one is used on a given route. Per-component barrel files are safe; directory-level mega-barrels are not.

## 7. Design System Integration

The visual language of The Data Specialist — lapis lazuli blues, gold fleck accents, and monospaced numerics — is expressed through a layered CSS custom property system.

### Token Hierarchy

```
src/styles/
├── variables.css   → Raw token values (hex codes, pixel values, font stacks)
├── tokens.css      → Semantic aliases referencing variables.css
├── globals.css     → Reset, @font-face declarations, base html/body styles
├── typography.css  → Heading, body, and code text style utilities
├── animations.css  → Keyframes and animation utility classes
└── themes/
    └── dark.css    → :root overrides for the dark theme (primary and default)
```

### How Components Consume Tokens

Components never reference raw values. They reference semantic tokens:

```css
/* Button.module.css */
.primary {
  background-color: var(--color-interactive);       /* resolves to #0D33A6 via tokens.css */
  color: var(--color-text-on-primary);              /* resolves to #FFFFFF */
}

.primary:hover {
  background-color: var(--color-interactive-hover); /* resolves to #3258A6 */
}

.accent {
  background-color: var(--color-accent);            /* resolves to #D9AE89 */
  color: var(--color-text-inverse);                 /* resolves to #141A26 */
}
```

### Token Resolution Chain

```
Component CSS Module
  └── references var(--color-interactive)
        └── defined in tokens.css as var(--color-primary-deep)
              └── defined in variables.css as #0D33A6
```

### Theme Switching

The dark theme is activated by the `data-theme="dark"` attribute on `<html>`, applied by `useTheme`. `src/styles/themes/dark.css` contains a single `:root` block that overrides the variables defined in `variables.css`. Switching themes is a DOM attribute change — no component re-renders, no class toggling on individual elements.

### Design Token Sync Workflow

Design token JSON files exported from Figma (`design/exports/tokens/*.json`) map 1:1 to CSS custom property names. A build script (invoked via `npm run tokens:sync`) reads the JSON exports and regenerates `variables.css`, keeping the CSS and Figma design system in sync without manual copy-paste.

## 8. Performance Considerations

### Route-Level Code Splitting

Every page component is lazy-loaded. In `App.tsx`:

```tsx
const Home = React.lazy(() => import('@/pages/Home'));
const Portfolio = React.lazy(() => import('@/pages/Portfolio'));
const ProjectDetail = React.lazy(() => import('@/pages/ProjectDetail'));
```

Wrapped in `<Suspense fallback={<PageSkeleton />}>`. Each page becomes its own chunk in the Vite build output. The initial bundle contains only the `Layout`, `Navbar`, and `Footer` components — everything else downloads on demand when the user navigates to that route.

**Target:** Initial JavaScript bundle under 150KB gzipped. Each page chunk under 50KB gzipped.

### Memoization Strategy

Memoization is applied **selectively**, not globally. The cost of unnecessary memoization (wasted memory, incorrect dependency arrays, developer confusion) often exceeds the cost of the extra render it was meant to prevent.

| API | Apply when |
|---|---|
| `React.memo` | Component re-renders frequently from parent; component is visually expensive (e.g., `DataTable` with 50+ rows); props are stable references |
| `useMemo` | Computation is genuinely expensive (e.g., filtering/sorting a large `Project[]` array); result is a reference type passed as prop to a memoized child |
| `useCallback` | Callback is passed as prop to a `React.memo`-wrapped child; callback is in the dependency array of a `useEffect` that must not re-run on every render |

**Rule:** Never wrap every component in `React.memo` or every value in `useMemo` by default. Profile first; memoize second.

### Asset Optimisation

- **Fonts:** Self-hosted `.woff2` only. Declared with `font-display: swap`. Subset to Latin Unicode range. No third-party DNS lookups.
- **Images:** `<img>` elements use `loading="lazy"` for all below-the-fold images. `srcset` and `sizes` attributes provide responsive variants. `width` and `height` attributes are always set to prevent CLS.
- **Icons:** SVG icons imported as React components via `@svgr/rollup`. Only icons used in the rendered tree are included in the bundle — no icon font sprites.
- **CSS:** Vite extracts all CSS Modules into a static `.css` file per chunk at build time. No runtime style injection.

### Bundle Output Structure

```
dist/
├── index.html
└── assets/
    ├── index.[hash].js        ← app shell + React runtime (~80KB gzip)
    ├── index.[hash].css       ← all global and shared component styles
    ├── Home.[hash].js         ← Home page chunk
    ├── Portfolio.[hash].js    ← Portfolio page chunk
    └── ...                    ← one chunk per lazy-loaded route
```

Run `npm run build:analyze` to open the Rollup bundle visualiser and inspect chunk composition.

## 9. Environment Variables

All environment variables are prefixed with `VITE_` and accessed via `import.meta.env.VITE_*`. Non-prefixed variables are not exposed to the browser bundle by Vite's build system.

Type declarations are added to `src/vite-env.d.ts`:

```ts
interface ImportMetaEnv {
  readonly VITE_CONTACT_ENDPOINT: string;
  readonly VITE_ANALYTICS_ID: string;
  readonly VITE_SITE_URL: string;
}

interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

### Variable Inventory

| Variable | Used in | Purpose |
|---|---|---|
| `VITE_CONTACT_ENDPOINT` | `src/services/contact.service.ts` | Contact form POST URL |
| `VITE_ANALYTICS_ID` | `src/services/analytics.service.ts` | Analytics provider tracking ID |
| `VITE_SITE_URL` | `src/config/seo.ts` | Canonical base URL for Open Graph tags |

### Environment Files

| File | Committed | Purpose |
|---|---|---|
| `.env.example` | ✅ Yes | Template with placeholder values. Documents all required variables for onboarding. |
| `.env` | ❌ No | Local development values. Listed in `.gitignore`. Never committed. |
| `.env.production` | ❌ No | Production values. Injected by CI/CD at build time. Never committed. |

Services that consume environment variables validate their presence at startup:

```ts
const ENDPOINT = import.meta.env.VITE_CONTACT_ENDPOINT;
if (!ENDPOINT) throw new Error('[contact.service] VITE_CONTACT_ENDPOINT is not defined');
```

### TypeScript Strict Mode

`tsconfig.app.json` enables `"strict": true`, activating:

- `strictNullChecks` — `null` and `undefined` are not assignable to other types without explicit handling.
- `noImplicitAny` — every value must have an explicit or inferable type.
- `noUnusedLocals` — unused variables are compile errors.
- `noUnusedParameters` — unused function parameters are compile errors.
- `exactOptionalPropertyTypes` — optional props cannot be assigned `undefined` explicitly unless declared as `T | undefined`.

The `"build"` script runs `tsc --noEmit` before `vite build`. A TypeScript error blocks the production build. Nothing ships with type errors.

### ESLint Rule Sets

`.eslintrc.cjs` applies:

- `@typescript-eslint/recommended` — TypeScript-aware linting.
- `react-hooks/recommended` — enforces Rules of Hooks and exhaustive dependency arrays.
- `jsx-a11y/recommended` — accessibility linting (ARIA attributes, focus management, semantic HTML).
- `import/no-restricted-paths` — enforces unidirectional architecture as described in §3.
- Custom: `no-console` (warn in development, error in CI), `no-debugger` (error), `@typescript-eslint/no-explicit-any` (error).

## 11. Testing Strategy

### Test Runner Configuration

Vitest is configured in `vitest.config.ts`:

- `environment: 'jsdom'` — simulates a browser DOM in Node.js for component rendering.
- `globals: true` — `describe`, `it`, `expect`, `beforeEach` are available without import in every test file.
- `setupFiles: ['src/tests/setup.ts']` — runs before every test suite; imports `@testing-library/jest-dom` to add DOM matchers globally and configures `window.matchMedia` and other browser API stubs.
- `coverage.provider: 'v8'` — native V8 coverage, faster than Babel instrumentation.

### Unit Tests (Vitest)

Unit tests cover code isolated from the DOM and other modules:

| Target | What to assert |
|---|---|
| `src/utils/` functions | Pure function input/output correctness for all edge cases |
| `src/hooks/` | Hook return values and state transitions via `renderHook` |
| `src/services/` | API call logic, error handling, and response parsing with mocked `fetch` |
| `src/components/ui/` | Rendering, prop variants, event firing, ARIA attributes, keyboard interaction |
| `src/data/` schemas | Zod schema accepts valid data and rejects invalid data with correct errors |

Example:

```ts
// src/hooks/useFilteredProjects.test.ts
import { renderHook } from '@testing-library/react';
import { useFilteredProjects } from '@/hooks/useFilteredProjects';
import { projects } from '@/data/projects';

it('returns only AI Engineering projects when that filter is active', () => {
  const { result } = renderHook(() =>
    useFilteredProjects(projects, 'AIEngineering')
  );
  expect(result.current.every(p => p.category === 'AIEngineering')).toBe(true);
});
```

### Integration Tests (Vitest + React Testing Library)

Integration tests render complete page sections and assert composed behaviour:

| Target | What to assert |
|---|---|
| `Portfolio` page | Filter bar and project grid interact correctly end-to-end |
| `Contact` page | Form validation, service call on valid submit, error state on failure |
| `Navigation` component | Dropdown opens/closes; keyboard navigation moves focus correctly |

Integration tests mock at the service boundary (mock `submitContactForm`, not `fetch`) to keep tests fast and isolated from network concerns.

### End-to-End Tests (Playwright)

E2E tests run against the production build in real browsers (Chromium, Firefox, WebKit):

| Spec | What to assert |
|---|---|
| `navigation.spec.ts` | All nav links navigate to the correct page; mobile menu opens and closes |
| `project-filtering.spec.ts` | Filter buttons update the visible project cards; URL query param updates |
| `contact-form.spec.ts` | Form submits, success message appears, error state appears on failure |

E2E tests run in CI against a `vite preview` server. They do not mock services — they test the full user journey from browser interaction to HTTP response (network-level mocking via Playwright's `route` API where needed).

### Coverage Targets

| Layer | Target | Rationale |
|---|---|---|
| `src/utils/` | 100% | Pure functions are trivially testable; 100% is the correct expectation. |
| `src/hooks/` | 90%+ | Hooks contain shared logic; high coverage prevents regressions across all consumers. |
| `src/services/` | 90%+ | Service functions are integration boundaries; covering error paths is essential. |
| `src/components/ui/` | 80%+ | Primitives are reused everywhere; regressions have wide blast radius. |
| `src/pages/` | 60%+ | E2E tests cover the user journeys that unit tests cannot; 60% covers the logic not reached by E2E. |

Coverage is reported on every CI run via `vitest run --coverage`. Pull requests that reduce coverage below target are flagged. Coverage is a signal and a guide — not an automated merge gate.
