# AI Prompt Engineering Master Kit
## Portfolio · Blog · Calculator Hub
### Stack: Next.js 15 · React 19 · TypeScript · Tailwind · shadcn/ui · Django 5 · Wagtail · Django Ninja

---

## SECTION 1 — VISION CONFIRMATION

You are building a **three-surface, API-first professional website** that functions as a developer brand platform. The **portfolio surface** is a curated, conversion-optimised showcase of your work (SSG/ISR, near-zero JS, schema.org markup). The **blog surface** is a full-featured publishing system managed entirely through Wagtail's headless CMS — with categories, tags, search, comments, RSS, and every SEO signal baked in from day one. The **Calculator Hub** is a collection of interactive, client-side tools that drive organic search traffic through long-tail keywords and shareable utility. The entire system is decoupled: a versioned REST API (auto-documented via OpenAPI/Swagger from Django Ninja) connects a Next.js 15 App Router frontend to a Django 5 + Wagtail backend, each deployed and scaled independently. SEO is not an afterthought — it is a first-class architectural constraint woven into every layer: rendering strategy, metadata API, structured data, canonical URLs, sitemaps, Core Web Vitals, and Wagtail's CMS-native SEO fields. AI assistance accelerates every phase of development using the five reusable prompt templates below.

### PROMPT #1 — PROJECT STRUCTURE MASTER

> **Strategic Power:** This prompt front-loads the single most critical architectural decision — folder layout — before a line of code is written. A bad structure compounds exponentially as the project grows. By anchoring the AI with your exact stack constraints, non-negotiable patterns (App Router colocation, Django app separation), and the requirement for annotated rationale, you get a structure you can actually defend and maintain. It also generates the scaffolding commands you can run immediately, eliminating the blank-canvas paralysis at project start.

---

### PROMPT #2 — DEEP RESEARCH AGENT

> **Strategic Power:** Most AI code generation fails because it starts from stale training data without verifying current library versions, deprecated APIs, or recent breaking changes. This prompt forces the AI to research before it generates — applying a seven-rule research protocol (Analyze → Structure → Include → Enforce → Embed → Prioritize → Ground) that mirrors how a senior engineer would approach an unfamiliar integration. The structured output (findings + warnings + corrections + next-step directives) feeds directly into Prompts #3 and #4 as a verified research brief, dramatically reducing hallucinations in generated code. The 2025–2026 recency filter is critical because Wagtail headless, Django Ninja, and Next.js 15 all changed significantly in the last 18 months.

---

```

You are an elite technical research analyst and full-stack architect specialising in

documentation and are up to date through 2026.

### PROMPT #3 — DJANGO + WAGTAIL BACKEND CODE GENERATOR

> **Strategic Power:** Backend generation prompts fail when they lack specificity about three things: the exact ORM pattern (Wagtail's Page tree vs standard Django models), the API contract (Ninja schemas, not DRF serializers), and the non-functional requirements (caching, security headers, ISR webhook signalling). This template makes all three explicit up front. The "chain-of-thought before code" directive prevents the AI from jumping straight to syntax — it must first reason about the data model, which catches relational design errors before they become migration debt. The strict output format (models → schemas → routers → signals → tests — always in this order) means you can paste sections directly into your repo structure.

---

### PROMPT #4 — NEXT.JS FRONTEND CODE GENERATOR

> **Strategic Power:** Frontend generation prompts almost always produce client-heavy code because the AI defaults to patterns from its training data (mostly React SPA articles). The "classify before code" directive catches the most expensive mistake in Next.js 15 development: adding `"use client"` to components that should be server-rendered, which inflates JS bundles and breaks SSG/ISR.

### PROMPT #5 — API INTEGRATION & DATA FLOW AGENT

> **Strategic Power:** The gap between "working backend" and "working frontend" is almost always a wiring problem: mismatched field names, missing authentication headers, wrong cache tags, broken error boundaries, or race conditions. This prompt is the bridge — it takes concrete code from both sides (Prompts #3 + #4) and generates the exact integration layer. By mandating that it operate *only* from actual code in the attachments (never from assumptions), it eliminates the most common AI error in integration tasks: inventing field names, endpoint URLs, or auth patterns that don't exist in the real codebase. The ISR revalidation webhook pattern and the typed API client pattern it enforces are both production-critical and commonly missed.

---

```

    error response shapes)
  - TypeScript-first API clients (openapi-typescript, Zod response validation)
  - Redis caching on both the Django side and Next.js fetch cache
  - Error handling across the full HTTP stack (network errors, 4xx, 5xx, timeouts)
  - Real-time UI patterns (optimistic updates, streaming, Suspense boundaries)
  - Security: CORS headers, JWT Bearer tokens, CSRF for mutations, rate-limit retries

OBJECTIVE
Generate ALL code required to connect a specific feature's frontend and backend, so
that data flows correctly from the Django/Wagtail database → API → Next.js page →

### 4A — CRITICAL FRONTEND AREAS

| Area | Why It Matters for This Project | Non-Negotiable Implementation |
|---|---|---|
| **Rendering strategy per route** | Wrong strategy = slow pages or stale content. Calculator Hub needs CSR; blog needs ISR; portfolio needs SSG. | Define `export const revalidate = N` or `export const dynamic = 'force-static'` at the top of every `page.tsx` |
| **`generateMetadata()` on every page** | Next.js metadata API is the primary mechanism for title, description, OG, Twitter, canonical — crawlers see this in the HTML `<head>` | Every `page.tsx` exports `generateMetadata()`. No exceptions. Dynamic routes resolve metadata from the API. |
| **Server Components as default** | Client Components ship JS to the browser, inflating LCP and hurting Core Web Vitals. Server Components render at build/request time and ship zero JS. | Only add `"use client"` when: `useState`, `useEffect`, `useRef`, event handlers, or browser APIs are used |
| **`next/image` for every image** | `<img>` tags cause layout shift (CLS), block rendering (LCP), and waste bandwidth. `next/image` handles lazy loading, responsive sizes, AVIF/WebP conversion. | Replace every `<img>` with `<Image>`. Set `priority={true}` on the above-fold image (hero image, post hero). Set explicit `width` + `height`. |
| **`next/font`** | Web font loading without `next/font` causes FOUT (Flash of Unstyled Text) = CLS penalty. | Load all fonts in `app/layout.tsx` with `next/font/google`, pass as CSS variable to Tailwind |
| **Structured data (schema.org)** | Google uses JSON-LD to understand page type, author, date, and rating — enabling rich snippets. | `Article` schema on all blog posts. `Person` schema on homepage/about. `SoftwareApplication` or `WebApplication` on each calculator. `BreadcrumbList` on nested pages. |
| **Sitemap + robots.txt** | Without a sitemap, Google may not discover all pages. Without robots.txt, the Wagtail admin may be crawled. | `app/sitemap.ts` (dynamic, pulls from API). `app/robots.ts` (disallows `/cms/`, `/api/`, `/admin/`). Submit sitemap to Google Search Console on launch. |
| **RSS Feed** | RSS is still indexed by Google. Tech blog readers use it. It signals content freshness. | `app/blog/feed.xml/route.ts` Route Handler returning valid RSS 2.0 XML using the `feed` npm package. |
| **Tag-based ISR revalidation** | Time-based ISR means new posts take up to N seconds to appear. Tag-based revalidation makes new posts appear within seconds of publishing. | Wagtail `page_published` signal → httpx webhook to `/api/revalidate` → `revalidateTag("blog-posts")` + `revalidateTag("blog-post-{slug}")` |
| **Core Web Vitals** | LCP < 2.5s, CLS < 0.1, INP < 200ms are Google ranking signals. | Run Lighthouse CI in GitHub Actions. Fail the build if any score drops below 85. Use Partial Prerendering (PPR) on pages mixing static shell + dynamic data. |

| Area | Why It Matters | Implementation |
|---|---|---|
| **Wagtail SEO fields on every Page** | Wagtail's `Page` base class includes `seo_title` and `search_description`. ### 4D — CUSTOM vs TEMPLATE-BASED IMPLEMENTATION

| Component | Recommendation | Reason |
|---|---|---|
| **UI components** (buttons, inputs, cards) | ✅ shadcn/ui (template-based) | Production-quality, accessible, dark-mode-ready. Custom components add zero SEO/performance value and cost weeks. |
| **Blog post body rendering** | ✅ Custom | Wagtail StreamField outputs JSON. You need custom renderers per block type. Use a `StreamFieldRenderer` pattern, not a generic library. |
| **Rich text rendering** | 🔶 Hybrid | Use `@portabletext/react` or `wagtail-richtext` npm package to parse Wagtail's HTML, but write custom renderers for each element (h2, blockquote, code). |
| **SEO metadata** | ✅ Custom | Use Next.js `generateMetadata()` API — never a third-party SEO library. Native API + no dependency overhead. |
| **OG image generation** | ✅ Custom (`next/og`) | `ImageResponse` from `@vercel/og` is the correct approach. Template-based OG image services add latency and external dependencies. |
| **Sitemap** | ✅ Custom (`app/sitemap.ts`) | Native Next.js API — dynamic, typed, always up-to-date. No `next-sitemap` package needed. |
| **Authentication** | 🔶 Context-dependent | If you need user login for a dashboard, use `next-auth v5`. If your site is read-only public, skip auth on the frontend entirely — JWT only on the Wagtail admin. |
| **Comments** | 🔶 Custom or Giscus | For blog comments: either implement a simple moderated comment model in Django (custom) or embed Giscus (GitHub Discussions-based, zero backend needed). Custom gives you full SEO control; Giscus is 2 hours vs 2 days. |
| **Analytics** | ✅ Template (Plausible/GA4) | Use Plausible (privacy-first, lightweight, no cookie banner) or GA4. Never build custom analytics. |
| **Search** | ✅ Custom | Expose Wagtail's PostgreSQL full-text search via a Django Ninja endpoint. Never use a client-side search library for SEO-indexed content — server-side search creates indexable result pages. |
| **Newsletter** | ✅ Template (Resend/Mailchimp) | Integrate Resend API for transactional emails. Custom email infrastructure is a production liability. |
| **CMS admin** | ✅ Wagtail (template-based) | Never build a custom CMS. Wagtail's admin is battle-tested and highly customisable without modification. |
| **Calculator logic** | ✅ Custom hooks | Each calculator is a pure function wrapped in a React hook (`useMortgageCalculator`, `useBMICalculator`). Zero dependencies, fully testable, easily extended. |
