## Chain-of-Thought Pre-Work

### A) Every URL the site needs

| URL | Rendering | Notes |
|-----|-----------|-------|
| `/` | SSG | Never changes without deploy |
| `/about` | SSG | Static bio, timeline, skills |
| `/portfolio` | ISR 3600 | Grid changes when you add work |
| `/portfolio/[slug]` | ISR 3600 | Case study per project |
| `/blog` | ISR 300 | Index refreshes as posts publish |
| `/blog/[slug]` | ISR 600 + tag-invalidation | Tag webhook from Django signal |
| `/blog/category/[category]` | ISR 300 | Category archive |
| `/blog/tag/[tag]` | ISR 300 | Tag archive |
| `/blog/feed.xml` | Route Handler | application/xml, no caching |
| `/tools` | SSG | Tools Hub listing |
| `/tools/[slug]` | SSG shell + CSR | Static SEO wrapper, client logic |
| `/contact` | SSR | Form needs fresh CSRF awareness |
| `/privacy` | SSG | Legal copy |
| `/terms` | SSG | Legal copy |
| `/sitemap.xml` | Generated by sitemap.ts | Queries BE for dynamic slugs |
| `/robots.txt` | Generated by robots.ts | Disallows /cms/ |
| `/api/revalidate` | Route Handler POST | Tag-based ISR invalidation webhook |
| `/api/og` | Route Handler GET | Dynamic OG image (ImageResponse) |
| `/api/contact` | Route Handler POST | Rate-limited contact form |

### B) Every data domain → Django app

| Domain | Django App | Key Models |
|--------|-----------|-----------|
| Auth | `apps/auth` | User, Role, Permission (simplejwt token management) |
| Portfolio | `apps/portfolio` | Project, Skill, SkillCategory, Experience, Testimonial |
| Blog | `apps/blog` | BlogIndexPage, BlogPost (Wagtail pages), Category, Tag (Snippets), Comment |
| Tools Hub | `apps/tools` | Tool, ToolCategory, Feature, Guide, Review |
| Cross-cutting | `apps/core` | ContactMessage, NewsletterSubscriber, SiteMetadata |

## 3. Key Architectural Decisions

**Decision 1 — Route groups `(marketing)` and `(legal)` for layout isolation**
About + Contact share no layout with the blog (which has a sidebar), so isolating them in `(marketing)` prevents accidental sidebar inheritance; legal pages in `(legal)` get their own minimal layout with no nav clutter.

**Decision 2 — `blog/feed.xml/route.ts` path, not `app/api/blog/feed/route.ts`**
Placing the RSS handler at `/blog/feed.xml` matches the conventional URL that RSS readers and search engines expect; a path under `/api/` would require extra redirects and break subscription links.

**Decision 3 — `schemas.py` separate from `models.py` in every app**
Django Ninja Pydantic schemas form the public API contract; keeping them in a dedicated file prevents circular imports, makes the contract auditable without reading model code, and isolates the API surface from ORM-level changes.

**Decision 4 — `services.py` layer enforced in every app**
All ORM queries live in `services.py`, never in routers or views; this is the single enforcement point for `select_related`/`prefetch_related` patterns that prevent N+1 query bugs (flagged as the #2 pitfall in the roadmap).

**Decision 5 — `loading.tsx` colocated with every route segment**
Each route segment having its own `loading.tsx` enables per-route Suspense streaming; a single root `loading.tsx` would flash a full-page skeleton on every navigation including child-route transitions, which is visually worse.

**Decision 6 — Tool implementations under `implementations/` subdirectory**
Individual tool implementations are nested under `components/tools/implementations/` rather than at the top level because they are consumed exclusively by `tools/[slug]/page.tsx`; top-level placement would create naming ambiguity with the shared `ToolShell` and `ToolCard` components.

**Decision 7 — `api/v1/__init__.py` as the single router-mount point**
All five Ninja routers are mounted in one place; when v2 becomes necessary, `api/v2/` is created without modifying `api/v1/` — backward compatibility preserved by default.

**Decision 8 — `wagtail_hooks.py` per app, not a global hooks file**
Wagtail's `AppConfig.ready()` auto-discovers `wagtail_hooks.py` per installed app; a global `hooks.py` in `config/` would not be discovered and would require manual import hacks.

**Decision 9 — `content/` directory for MDX legal pages on the frontend**
Privacy Policy and Terms of Service are edited by hand as MDX files, not fetched from an API; committing them as MDX in `content/` means they version-control alongside code, not content — correct for legal copy that changes only on deliberate decision.

**Decision 10 — `CLAUDE.md` and `AGENTS.md` in both repos**
`CLAUDE.md` carries session-level Claude Code context (dev commands, key files); `AGENTS.md` carries the contract for all AI agents (stack, what to never modify); keeping both ensures consistent AI assistance across contributors and sessions.

### 6.4 Authentication Flow — JWT (6 Steps)

```
1. User POSTs credentials → /auth/login
2. Auth Service validates credentials against Auth DB
3. Auth Service returns signed Access Token (JWT)
4. Client stores token in an HttpOnly Cookie (no JS access, XSS-safe)
5. Client attaches token in Authorization: Bearer <token> header on subsequent requests
6. Downstream services validate the token via Auth Service introspection (or a shared JWKS endpoint)
```

| Endpoint | Purpose |
|----------|---------|
| `POST /auth/login` | Issue Access + Refresh tokens |
| `POST /auth/logout` | Invalidate refresh token |
| `POST /auth/token/refresh` | Exchange Refresh token for new Access token |
| `GET  /auth/me` | Return user profile from token claims |

### 7.2 CORS Bypass — Next.js Dev Proxy

In production the API Gateway handles CORS. Locally, the Next.js dev server is configured as a reverse proxy to eliminate CORS preflight errors during development:

```typescript
// next.config.ts
const nextConfig: NextConfig = {
  async rewrites() {
    return [
      {
        source: '/api/:path*',
        destination: 'http://localhost:8000/api/:path*', // Proxied to Django
      },
    ];
  },
};
```

The frontend always calls `/api/*` — in dev the rewrite silently forwards to `:8000`; in production the same path hits the real API Gateway. **No environment-specific fetch URLs needed in application code.**

## 8. Additional Architectural Decisions (Images Reference)

**Decision 11 — API Gateway as the single CORS and auth boundary**
All CORS policies and token-validation middleware live exclusively in the API Gateway; individual microservices trust only requests that have already passed through it. This means services never need CORS headers configured locally and cannot be accessed by external clients without going through the gateway — reducing the attack surface.

**Decision 12 — One database per service (no shared DB)**
Each microservice owns its PostgreSQL instance exclusively. Cross-service joins are forbidden; data that needs to be read across service boundaries is either duplicated (denormalised) into the consuming service's DB via an event/message, or fetched through the API. This eliminates tight schema coupling and allows independent migration cycles.

**Decision 13 — Next.js rewrites proxy for local dev (no per-environment fetch URLs)**
Using `next.config.ts` rewrites to proxy `/api/*` to `localhost:8000` means all application code references the same relative `/api/` path in every environment. Production traffic hits the real gateway at the same path — zero environment-specific branching in fetch logic.

**Decision 14 — Zustand for global client state, TanStack Query for server state**
`Zustand` manages truly global UI state (auth status, theme, modals) that persists across route navigations. `TanStack Query` manages server-derived data (blog posts, project lists) with its built-in caching and stale-while-revalidate semantics. Mixing them is intentional — they solve different problems and should not be collapsed into one store.

**Decision 15 — Auth Service extracted first in microservices migration**
Authentication is the highest-risk shared concern. Extracting it first as a standalone service with its own DB means all other services can depend on a stable, versioned JWT contract before any other migration work begins. It also forces the team to solve the gateway routing and inter-service JWT validation pattern once, at the start.

**Decision 16 — `djangorestframework-simplejwt` for local dev; shared JWKS in production**
In local development, each Django process validates JWTs independently using the shared `SECRET_KEY`. In production the Auth Service publishes a JWKS endpoint; all other services validate tokens by fetching the public keys from that endpoint — no `SECRET_KEY` sharing across service boundaries.

**Decision 17 — Docker Compose for local data layer; no local microservice isolation**
Running each future microservice in its own Docker container locally adds unnecessary startup friction during active development. All Django code runs as a single `runserver` process locally; Docker Compose is used only for the stateful dependencies (PostgreSQL, Redis) that cannot run as simple Python processes.

**Decision 18 — `/tools` replaces `/calculators` in the Tools Hub Service**
The production architecture diagram labels the fourth service *Tools Hub* (with endpoints for tools, categories, features, guides, reviews), which is broader than the initial *Calculators* concept. When extracting `apps/calculators` into a standalone service, rename the app to `apps/tools` and extend the model set accordingly. The frontend route `/calculators` can be redirected to `/tools` via a `next.config.ts` redirect to preserve existing URLs.

### 9.6 Authentication Flow — Local Dev vs Production Comparison

Both images show a JWT authentication flow. The two differ in scope:

**Local Dev (Image 2 — 3 Steps)**

```
1. React client POSTs credentials → /api/token/
   (Next.js proxy forwards to localhost:8000/api/token/)
2. Django DRF (simplejwt) validates and returns Access + Refresh JWTs
3. React client stores tokens in HttpOnly cookies;
   attaches Access Token in Authorization: Bearer <token> on all subsequent requests
```

**Production (Image 1 — 6 Steps)**

```
1. User POSTs credentials → /auth/login  (via API Gateway)
2. Auth Service validates credentials against Auth DB
3. Auth Service returns signed Access Token (JWT)
4. Client stores Access Token in HttpOnly Cookie (XSS-safe)
5. Client attaches token in Authorization: Bearer <token> header on subsequent requests
6. Downstream services validate token via:
   a. Auth Service introspection call (simple but adds latency), OR
   b. JWKS endpoint (Auth Service publishes public keys; services validate locally — preferred)
```

**HttpOnly Cookie note:** Both flows store tokens in HttpOnly cookies. The cookie is set by the *backend*, not by JavaScript — `document.cookie` cannot read it. This means `authStore.ts` (Zustand) must infer authentication state from a lightweight `/auth/me` endpoint response on app load, not by reading the token directly.

### 9.9 New Architectural Decisions (from Images)

**Decision 19 — Wagtail Admin is excluded from the API Gateway's public routing**
The Wagtail admin UI (`/cms/`) serves HTML, sessions, and file uploads — none of which fit the API Gateway's JSON-oriented request/response contract. In production, `/cms/` should be routed directly to the Blog Service via a separate internal load-balancer rule or VPN tunnel, never exposed through the public API Gateway. The `robots.ts` already disallows `/cms/` from crawlers; IP allowlisting should be added at the infrastructure level.

**Decision 20 — JWKS-based token validation is mandatory by Phase 3**
Once Auth is an independent service (Phase 2), all other services must validate JWTs using the Auth Service's JWKS endpoint (`/auth/.well-known/jwks.json`), not by sharing the `SECRET_KEY`. Sharing a secret across independent processes breaks the isolation guarantee of the microservices architecture. Cache JWKS responses with a 5-minute TTL to avoid per-request latency.

**Decision 21 — Blog Service keeps Wagtail's `page_published` signal for ISR invalidation**
The `page_published` Wagtail hook fires a Django signal whenever a blog post is published or updated. A `wagtail_hooks.py` receiver in `apps/blog/` makes an authenticated POST to `NEXT_PUBLIC_REVALIDATE_URL/api/revalidate` with the affected slug's cache tags. This preserves the ISR invalidation webhook already described in Section 1 (`/api/revalidate/route.ts`) without any additional infrastructure.

**Decision 22 — Docker Compose Redis is session store for Wagtail admin in local dev**
Wagtail's admin session is stored in Django's session framework. In local dev, `SESSION_ENGINE = 'django.contrib.sessions.backends.cache'` backed by the Docker Compose Redis instance. This ensures admin sessions persist across `runserver` restarts without needing the database session backend (slower) or file-based sessions (not portable across team members).

**Decision 23 — Internal service communication always proxied through API Gateway; no direct service-to-service HTTP calls**
Image 1 shows internal communication as green dashed arrows going *through* the gateway. Services must not maintain a hard-coded URL to another service's internal address. All service-to-service reads (e.g., Blog Service enriching a response with author data from Auth Service) must go through the API Gateway's internal routing layer. This enforces a single audit and rate-limit point for all traffic, internal or external.

**Decision 24 — `SiteLinksSearchBox` schema on `/tools` is the only proactive schema.org enhancement for CSR pages**
All other Schema.org types (`Person`, `Article`, `BreadcrumbList`, `SoftwareApplication`) are injected server-side via the `<JsonLd />` component in Server Components — they are present in the initial HTML payload and visible to crawlers. `/tools/[slug]` is a CSR shell, so its `SoftwareApplication` schema must be injected in the `<head>` from the static SSG shell (not after hydration) to ensure Googlebot sees it without JavaScript execution.

### 10.4 ISR Invalidation Signals — `apps/portfolio` and `apps/tools`

Both new `signals.py` files follow the identical pattern already used in `apps/blog/signals.py`:

```python
# apps/portfolio/signals.py
from django.dispatch import receiver
from wagtail.signals import page_published, page_unpublished
from .models import ProjectPage
import requests, os

def _revalidate(tag: str):
    url = os.environ.get("NEXTJS_REVALIDATE_URL", "")
    secret = os.environ.get("REVALIDATE_SECRET", "")
    if url and secret:
        requests.post(
            f"{url}/api/revalidate",
            json={"tag": tag},
            headers={"x-revalidate-secret": secret},
            timeout=5,
        )

@receiver(page_published, sender=ProjectPage)
@receiver(page_unpublished, sender=ProjectPage)
def on_project_page_change(sender, instance, **kwargs):
    _revalidate("portfolio")
    _revalidate(f"project-{instance.slug}")
```

```python
# apps/tools/signals.py  — identical structure
from .models import ToolPage

@receiver(page_published, sender=ToolPage)
@receiver(page_unpublished, sender=ToolPage)

    _revalidate("tools")
    _revalidate(f"tool-{instance.slug}")
```

### 10.5 SEO Prioritisation — `/tools` and `/tools/[slug]`

#### Rendering strategy change

| Path | Before | After | Reason |
|------|--------|-------|--------|
| `/tools` | `[SSG]` | `[ISR:3600]` | Wagtail publishes new tools → invalidation webhook must be able to revalidate the index |
| `/tools/[slug]` | `[SSG] shell + [CSR]` | `[ISR:3600] shell + [CSR]` | `page_published` signal fires `revalidateTag("tool-{slug}")` — SSG cannot respond to that |

The ISR shell renders all static SEO content at request time (or from cache); the CSR layer hydrates tool-specific interactive logic after paint. This preserves Google's ability to index the full SEO payload without JavaScript.

#### `src/app/tools/page.tsx` — ISR with tag

```typescript
// src/app/tools/page.tsx
import { fetchTool } from "@/lib/api/tools";

export const revalidate = 3600;  // ISR fallback; webhook revalidates sooner on publish

export default async function ToolsPage() { /* ... */ }
```

#### `src/app/tools/[slug]/page.tsx` — ISR shell, schema in server payload

```typescript
// src/app/tools/[slug]/page.tsx
import { fetchTool } from "@/lib/api/tools";
import { JsonLd } from "@/components/layout/JsonLd";

export const revalidate = 3600;

export async function generateMetadata({ params }) {
  const tool = await fetchTool.detail(params.slug, { next: { tags: [`tool-${params.slug}`] } });
  return {
    title: tool.seo_title || tool.title,
    description: tool.search_description || tool.short_description,
    alternates: { canonical: `/tools/${tool.slug}` },
  };
}

export default async function ToolPage({ params }) {
  const tool = await fetchTool.detail(params.slug, { next: { tags: [`tool-${params.slug}`] } });
  return (
    <>
      {/* schema_org_json is authored in Wagtail admin; injected server-side — visible to Googlebot */}
      {tool.schema_org_json && <JsonLd data={tool.schema_org_json} />}
      <ToolShell tool={tool} />
      {/* CSR layer loads beneath — interactive logic only, no SEO-critical content */}
    </>
  );
}
```

`schema_org_json` is **never hydrated or generated client-side**. It is fetched from `ToolPage.schema_org_json` (PostgreSQL) at ISR render time and written into the HTML response as a `<script type="application/ld+json">` tag. Googlebot sees it on first HTML parse.

#### `src/lib/api/tools.ts` — cache tag passthrough

```typescript
// src/lib/api/tools.ts
export const fetchTool = {
  list: (opts = {}) =>
    apiFetch<ToolListOut[]>("/api/v1/tools/", { next: { tags: ["tools"], revalidate: 3600 }, ...opts }),

  detail: (slug: string, opts = {}) =>
    apiFetch<ToolOut>(`/api/v1/tools/${slug}/`, { next: { tags: [`tool-${slug}`], revalidate: 3600 }, ...opts }),
};
```

#### `/api/revalidate/route.ts` — tag routing (confirm `"tools"` and `"portfolio"` are handled)

```typescript
// src/app/api/revalidate/route.ts  — ensure tags are whitelisted
const VALID_TAG_PREFIXES = ["blog-", "post-", "tools", "tool-", "portfolio", "project-"];

export async function POST(req: Request) {
  const { tag } = await req.json();
  if (!VALID_TAG_PREFIXES.some(p => tag.startsWith(p))) {
    return Response.json({ error: "unknown tag" }, { status: 400 });
  }
  revalidateTag(tag);
  return Response.json({ revalidated: true, tag });
}
```

### 10.8 New Decisions (Section 10)

**Decision 25 — Wagtail `promote_panels` is the canonical source for tool and project SEO fields**
Wagtail's built-in `promote_panels` (SEO title, meta description, slug) is used for every `ProjectPage` and `ToolPage`. The frontend's `generateMetadata()` reads `seo_title` and `search_description` from the API response — these fields are never hardcoded in Next.js. This means an editor can fix a missing meta description without a code deploy.

**Decision 26 — `schema_org_json` on `ToolPage` is editor-owned, not code-generated**
The `SoftwareApplication` JSON-LD for each tool is stored in `ToolPage.schema_org_json` (a `JSONField` edited in Wagtail admin) rather than generated programmatically from model fields. This allows per-tool customisation (different `applicationCategory`, `offers`, `featureList`) without code changes. A Wagtail panel with `help_text` provides an inline example payload to guide editors.

**Decision 27 — `/tools/[slug]` ISR shell must contain zero CSR-dependent SEO content**
The ISR shell component (`ToolShell`) must be split so that all content Googlebot needs — title, description, `JsonLd`, breadcrumb schema — is rendered in the async Server Component. The `'use client'` calculator or interactive tool component is a child mounted below the fold. No SEO content may depend on `useEffect`, `useState`, or any client-only data fetch.

**Decision 28 — `generateStaticParams` is retained for `/tools/[slug]` despite ISR**
Even with ISR, `generateStaticParams` is called at build time to pre-render the most important tool pages as static HTML. Pages not in `generateStaticParams` are rendered on first request and then cached. This gives fast TTFB for the top tools while still supporting new tools added via Wagtail without a rebuild.

```typescript
// src/app/tools/[slug]/page.tsx
export async function generateStaticParams() {
  const tools = await fetchTool.list();
  return tools.map(t => ({ slug: t.slug }));
}
```
